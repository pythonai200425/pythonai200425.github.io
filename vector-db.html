CHAPTER 1 — INTRODUCTION

What Is a Vector Database?

A vector database stores data as numerical arrays (vectors) that capture meaning — not just exact text.

💡 Think of it like a library organized by theme instead of alphabetical order.

In a traditional database, querying for WHERE name = "Alice" returns only exact matches. A vector database, by contrast, understands that "happy" is close to "joyful" and "glad" — because these words share similar vector representations in high-dimensional space.

🗄️

Traditional DB

Matches exact values: WHERE name = "Alice"

Fast for structured, exact lookups. SQL, B-Tree indexes, rows and columns.

🧠

Vector DB

Finds similar meanings: "happy" → "joyful", "glad"

Powered by high-dimensional embeddings and approximate nearest neighbor search.

CHAPTER 2 — EMBEDDINGS

From Text to Vectors

An embedding is a model that converts text (or images) into a list of numbers — a vector — where similar things land close together in space.

Each dimension captures a different aspect of meaning — sentiment, topic, style — learned by the model from massive text data.

Why vectors?

Vectors let us use linear algebra to measure semantic closeness. The angle between two vectors encodes how "similar" two pieces of text are in meaning.

What models produce them?

sentence-transformers (free, local), OpenAI Embeddings API, Cohere Embed, and Google Vertex AI all produce embeddings from text.

CHAPTER 3 — SIMILARITY SEARCH

How Does Similarity Work?

Vector similarity measures how "close" two vectors are in high-dimensional space. The two most common metrics are Cosine Similarity and Euclidean Distance.

📐

Cosine Similarity

Measures the angle between vectors. Ignores length — great for text. Range: -1 to 1. A score of 1 means identical direction (same meaning). Most vector DBs use this for text.

📏

Euclidean Distance

Measures straight-line distance between points. Intuitive "as the crow flies" metric. Good for image similarity and normalized embeddings. Lower = more similar.

💡 Most vector DBs use cosine similarity for text — it captures direction of meaning, not vector magnitude.

🧮 Cosine Similarity Calculator

Cosine Similarity = cos(θ) = A · B ‖A‖ ‖B‖
Example: A = (1, 2)  ·  B = (2, 3)
Step 1 — Dot product:   A · B = (1×2) + (2×3) = 2 + 6 = 8 Step 2 — Magnitude A:   ‖A‖ = √(1² + 2²) = √5 ≈ 2.236 Step 3 — Magnitude B:   ‖B‖ = √(2² + 3²) = √13 ≈ 3.606 Step 4 — Similarity:    cos(θ) = 8 / (2.236 × 3.606) = 8 / 8.062 ≈ 0.9923 🟢 Very similar

CHAPTER 4 — COMPARISON

Traditional DB vs. Vector DB

Both database types have their strengths. Knowing when to use each — or both together — is key to designing modern AI systems.

Feature Traditional DB Vector DB
Query type Exact match, SQL filters Semantic similarity, KNN
Data stored Rows, columns, JSON High-dimensional vectors + metadata
Best for Transactions, structured data Search by meaning, AI pipelines
Index type B-Tree, Hash HNSW, IVF, Flat
Example use User accounts, orders RAG, recommendations, semantic search
Scale Billions of rows, mature tooling Millions-billions of vectors, rapidly evolving
Query language SQL (standardized) SDKs, REST APIs, no universal standard
ℹ️ Hybrid approach: Many systems use both — a Postgres DB for structured data + a vector DB for semantic search. Tools like pgvector even add vector support directly to Postgres.

CHAPTER 5 — CORE CONCEPTS

How Vector DBs Work Under the Hood

Understanding the internals helps you choose the right settings for your use case.

🗂️

Indexing

Pre-organizes vectors so searches are fast. Without an index, you'd scan every vector — slow at scale. Common index types: HNSW (best recall), IVF (clustered), Flat (exact but slow).

🔎

KNN / ANN Search

KNN = exact nearest neighbors (100% accurate, slow for large datasets). ANN = approximate (faster, slightly less precise). Trade speed vs. accuracy depending on your needs.

🏷️

Metadata Filtering

Combine vector search with filters: where={"author": "Alice"}. Narrows results by context — e.g., search only documents from this year, only PDFs, only high-rated items.

💾

Persistence

Vectors and metadata are stored on disk — surviving restarts. Some DBs also support cloud sync. In-memory mode (e.g. Chroma) is faster but data is lost on restart.

Common Index Types Explained

🕸️
HNSW
Best Recall · Fast

Hierarchical Navigable Small World. Builds a multi-layer graph — upper layers give a coarse overview, lower layers get precise. Search starts at the top and zooms in. Gives ~95–99% recall at 10–100× the speed of brute force. The go-to choice for most production systems.

⚡ Speed: Very fast  ·  🎯 Accuracy: Very high
🗂️
IVF
Clustered · Scalable

Inverted File Index. Groups vectors into clusters using k-means. At query time, only the nearest clusters are searched — skipping most of the dataset. Great for large datasets that won't fit entirely in RAM. Accuracy depends on how many clusters you probe (nprobe).

⚡ Speed: Fast  ·  🎯 Accuracy: Tunable
📋
Flat
Exact · Slow at Scale

Brute Force / Exact Search. Compares the query vector against every single stored vector. Zero approximation — 100% accurate results guaranteed. Fine for small datasets (<10K vectors) but impractical at scale. Often used as a benchmark to measure accuracy loss from other indexes.

⚡ Speed: Slow O(n)  ·  🎯 Accuracy: 100% exact
HNSW — Multi-Layer Graph

Search drills top→bottom through graph layers

IVF — Clusters (Voronoi)

Only nearest clusters are searched — rest are skipped

Flat — Brute Force (Every Vector)

Every vector compared — guaranteed exact result, O(n) cost

📊 How to pick:   Use Flat for <10K vectors or exact benchmarking.   Use IVF for large, memory-constrained datasets — tune nprobe for recall vs. speed.   Use HNSW for everything else — the default production recommendation.

CHAPTER 6 — TOOLS

Popular Vector Databases

Each vector database has a different tradeoff between ease of use, performance, and deployment model. Here are the four most popular options for Python developers.

🌲 Pinecone
Fully Managed

Cloud-native, fully managed service. No infrastructure to maintain. Great for production workloads. Scales automatically. Supports real-time upserts and metadata filtering.

Best for: Production apps · Teams that want zero ops
🕸️ Weaviate
Open Source

Open-source with GraphQL API. Can self-host or use Weaviate Cloud. Rich schema features, multi-tenancy, and hybrid search (vector + BM25 keyword). Python client available.

Best for: Complex schemas · Hybrid search · Self-hosting
⚡ FAISS
Python Library

Meta's library (Python + C++). Lightning-fast ANN search. No server — runs in-process. Supports GPU acceleration. Great for research and prototyping. Not a full DB — no metadata or persistence by default.

Best for: Speed · Research · Large-scale ANN · No server needed
🎨 Chroma
Lightweight

Lightweight, Python-friendly. Perfect for learning and small projects. In-memory or persistent mode. Built-in embedding support via sentence-transformers. Minimal setup — just pip install chromadb.

Best for: Learning · Prototypes · Small projects · LLM apps
Quick pick guide:
🎓 Learning → Chroma  |  🧪 Research → FAISS  |  🏗️ Self-host → Weaviate  |  🚀 Production SaaS → Pinecone

Also Worth Knowing

🐘 pgvector

Postgres extension that adds vector storage and similarity search. If you already use Postgres, this is the lowest-friction path to vector search. Supports L2, inner product, and cosine distance.

🔴 Qdrant

Rust-based, open source, high performance. Excellent Python SDK. Supports payload filtering, named vectors, and on-disk storage. Growing rapidly in the community.

🟣 Milvus

Enterprise-grade open-source vector DB. Designed for billion-scale workloads. Supports multiple index types (HNSW, IVF, DiskANN). Strong GPU support.

🦙 LanceDB

Columnar storage, serverless-friendly. Built on Apache Arrow. Zero-copy data access. Native Python and JavaScript SDKs. Great for embedded use cases.

CHAPTER 7 — HANDS-ON

Python Example with Chroma

Chroma makes it trivially easy to get started with vector search in Python. Here's a complete working example.

PYTHON
import chromadb
from sentence_transformers import SentenceTransformer

# Load embedding model
model = SentenceTransformer("all-MiniLM-L6-v2")

# Create client & collection
client = chromadb.Client()
collection = client.create_collection("docs")

# Add documents (auto-embedded)
collection.add(
    documents=["Cats are great pets",
               "Dogs are loyal companions"],
    ids=["doc1", "doc2"]
)

# Query
results = collection.query(
    query_texts=["I love animals"],
    n_results=2
)
print(results["documents"])
01

Embed

SentenceTransformer converts text → vector automatically. The model runs locally — no API calls needed.

02

Store

Chroma saves vectors + original text + unique IDs. You can also pass metadata dicts for filtering later.

03

Query

Pass a question — Chroma returns the most similar docs. Results include documents, distances, and metadata.

04

Filter

Add metadata filters: where={"author": "Alice"} to narrow results without sacrificing semantic power.

CHAPTER 8 — RAG

RAG: Retrieval-Augmented Generation

RAG connects an LLM to your own data via a vector DB — so it answers with facts, not just training memory.

The vector DB retrieves the most relevant document chunks, which are injected into the LLM's context window for accurate, sourced answers.

🧑
User Query
"What is HNSW?"
→
🗄️
Vector DB Search
Top-K chunks retrieved
→
📎
Inject Context
Chunks added to prompt
→
🤖
LLM Answer
Accurate, sourced reply

❓ Why RAG?

  • 🔒LLMs don't know your private data
  • 🎯Reduces hallucinations with real sources
  • 🔄Update knowledge without retraining
  • 💰Much cheaper than fine-tuning

🗄️ Vector DB's Role in RAG

  • 📦Stores your documents as embeddings
  • ⚡Retrieves top-K relevant chunks fast
  • 🏷️Filters by metadata (source, date, etc.)
  • 🔗Bridges your data and the LLM

RAG Pipeline in Python

PYTHON
import chromadb
import anthropic  # or openai

# 1. Set up vector DB with your docs
client = chromadb.PersistentClient(path="./docs_db")
collection = client.get_or_create_collection("my_docs")

# 2. Query — find relevant chunks
def rag_query(user_question: str) -> str:
    results = collection.query(
        query_texts=[user_question],
        n_results=3
    )
    context = "\n\n".join(results["documents"][0])

    # 3. Inject context into LLM prompt
    prompt = f"""Answer based on this context:
{context}

Question: {user_question}"""

    # 4. Get LLM answer
    claude = anthropic.Anthropic()
    response = claude.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=1024,
        messages=[{"role": "user", "content": prompt}]
    )
    return response.content[0].text
CHAPTER 9 — WRAP-UP

Best Practices, Pitfalls & Takeaways

Avoid common mistakes and follow proven patterns to build reliable vector search applications.

⚠️ Common Pitfalls

  • 🎭

    Bad Embeddings

    Wrong model = meaningless vectors. Match model to your domain. Code search needs a code-trained model, not a general text model.

  • ✂️

    Chunking Errors

    Too large = diluted meaning. Too small = lost context. Aim for 200–500 tokens with overlap between chunks.

  • 🕰️

    Stale Index

    Updated docs need re-embedding. Automate re-indexing with webhooks or scheduled jobs — stale vectors return wrong results.

  • 🎲

    No Metadata Filtering

    Pure vector search retrieves semantically similar content from ALL documents. Add metadata filters to scope results properly.

✅ Best Practices

  • 🚀

    Start with all-MiniLM-L6-v2

    Fast, solid, free to run locally. 384 dimensions. Great baseline for most text tasks before optimizing.

  • 📏

    Chunk at ~200–500 tokens

    For most text tasks. Use sliding window overlap (50–100 tokens) to avoid cutting context at chunk boundaries.

  • 🔗

    Combine vector + metadata filters

    Vector search finds semantically similar content; metadata filters scope it. Use both for precise, context-aware results.

  • ⚡

    Use ANN (HNSW) for large datasets

    Speed vs. accuracy trade-off: HNSW gives 95–99% recall at 10-100x speed. Only use Flat (exact) for <10K vectors.

✅ Key takeaway: Vector DBs power semantic search, RAG, and recommendations — essential tools for modern Python AI projects.
